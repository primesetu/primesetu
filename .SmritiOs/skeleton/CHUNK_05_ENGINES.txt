SMRITI-OS v3.0 — SKELETON CHUNK 05: REUSABLE ENGINE ARCHITECTURE
================================================================
Core Principle: "Build 10 engines — not 100 modules."
All ERP screens follow: Grid + Filters + Form + Search + Export + Permissions
Only FIELDS, WORKFLOWS, and BUSINESS RULES change between modules.

─── THE 5 CORE REUSABLE ENGINES ────────────────────────────────
Engine              | Component          | Config Points                | Benefit
Universal Grid      | <AppGrid />        | columns, APIs, filters, sort | One component all list screens
Universal Form      | <AppForm /> + JSON | fields array: name/type/val  | Auto-render inputs from schema
Universal CRUD      | FastAPI Router     | GET/POST/PUT/DELETE/LIST/EXPORT | Zero repeat code
Universal RBAC      | Permission MW      | view/create/edit/delete/export per role | Same check everywhere
Universal Report    | Recharts+SheetJS   | filters, export, print, dates | Uniform report experience

─── OLD WAY vs SMRITI-OS WAY ───────────────────────────────────
Old Way : Design UI → Forms → APIs → Validation → REPEAT per module
          Result  : Inconsistent UI, bugs per module, slow delivery

Smart Way: Build engines → Define JSON schema → Module auto-generated
           Result  : Consistent UI, rapid delivery, easy maintenance

─── SCHEMA EXAMPLE (Item Master as config) ──────────────────────

─── WHAT CANNOT BE REUSED (Custom Business Logic) ───────────────
Domain                    | Why Custom Code Required
POS / Billing Engine      | Discounts, multi-tax, rounding, returns, holds
Inventory Movement Engine | Stock in/out, inter-store, batch, FIFO/LIFO
GST Rules Engine          | CGST/SGST/IGST splits, HSN, e-invoice, RCM
Sync Engine               | Delta sync, conflict resolution, offline queue
Promotions/Pricing Engine | BxGy, tier pricing, loyalty redemption

─── ENGINE OVERRIDE STRATEGY (Layered) ──────────────────────────
Layer               | Description                            | Owner
Core Engine         | Universal — NEVER modified directly    | Architecture team
Module Config       | Schema/metadata defining standard beh  | Module developers
Module Overrides    | Hooks extending specific engine behav  | Module developers
Client Extensions   | Isolated per-client — outside core     | Implementation partners

─── AI-AUGMENTED WORKFLOW ───────────────────────────────────────
Owner   | Responsibility
Humans  | Architecture + business logic (GST rules, sync logic, pricing)
Schemas | Define structure (fields, types, validations, relationships)
AI      | Generate repetitive implementation from schemas at high speed
        | (forms, CRUD scaffolding, validation, API scaffolding, tests)

Rule: AI accelerates speed. Humans control correctness. Neither replaces the other.

─── ARCHITECTURAL STANCE ────────────────────────────────────────
Strategy: MODULAR MONOLITH FIRST
Evolve to service separation ONLY where operational scale requires it.
Avoids: premature microservices, distributed system overhead,
        unnecessary infrastructure cost.
