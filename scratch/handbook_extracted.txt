1.  Migration Philosophy
1.1  Inspired Schema Approach
SMRITI OS does not copy Shoper9's schema. It uses an Inspired Schema approach: field names and business logic are studied and preserved where practical, but the schema is redesigned from the ground up for PostgreSQL, multi-store cloud architecture, and offline sync. This eliminates legal risk and removes decades of technical debt from the legacy SQL Server single-store design.

1.4  The Sovereign Database Layer
SMRITI OS uses sovereign.py as an intelligent proxy layer. It queries legacy Shoper9 data natively via the existing SQL Server connection while simultaneously exposing modern PostgreSQL-backed metadata (image_url, description_html, IRN, sync columns). This enables zero-data-loss migration for existing clients while delivering modern capabilities immediately — and is the single most important architectural advantage for GTM with existing Shoper9 clients.

Query 9 — Nullable Column Audit (NEW)
Identifies nullable columns in critical tables that SMRITI OS enforces as NOT NULL. Every row found here needs a placeholder resolution strategy before migration begins.
Query 10 — Check Constraints — Business Rules in DB (NEW)
Check constraints encode business rules directly in the database. These must be identified and converted to FastAPI Pydantic validators. If not identified, they silently block inserts during migration.
Query 11 — Unique Constraints (NEW)
Unique constraints define what cannot have duplicates — barcode uniqueness, GSTIN uniqueness, BillNo uniqueness. If not recreated in PostgreSQL, silent data quality failures occur post-migration.

6.2  legacy_id Retention Policy (NEW)
legacy_id columns are retained for 90 days post go-live. After 90 days and successful Tally Sync validation, legacy_id columns are set to NULL. After 180 days they are dropped via an Alembic migration. This prevents them from becoming permanent technical debt while allowing sufficient time for cross-reference validation.
7.  Migration Execution Plan
7.1  Migration Waves
Tables must be migrated in wave order to respect FK dependencies. Never migrate a table before all its FK parent tables are migrated and validated.
7.2  Migration Script Pattern — With Idempotency Guard (v2.0)
Every migration script must include an idempotency check. If the script crashes mid-wave and is re-run, it must not create duplicate records.
