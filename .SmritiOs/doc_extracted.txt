8.
Reusable Engine Architecture
8.1
The Core Principle
Modern ERP architecture is built on a simple but powerful principle: do not build the same thing twice. When every module requires a separate form, grid, save logic, search, and validation UI built from scratch — the result is wasted time, inconsistent UI, compounding bugs, and a maintenance nightmare. Smriti OS avoids this trap through a Reusable Engine Architecture.
8.2
The Real Pattern of ERP Screens
Ninety percent of ERP screens follow the same pattern: a data grid, filters, a form for detail entry, and report output. Only the fields, workflows, and business rules change between modules. The UI structure is identical.
Module
Fields (Changes Per Module)
UI Structure (Same Everywhere)
Item Master
item_name, barcode, MRP, HSN, tax
Grid + Add Button + Form + Search + Filters + Export + Permissions
Customer Master
customer_name, phone, address, GST
Grid + Add Button + Form + Search + Filters + Export + Permissions
Supplier Master
supplier_name, GSTIN, credit days
Grid + Add Button + Form + Search + Filters + Export + Permissions
Conclusion: Only metadata changes between modules. The UI structure is identical. The correct architecture is to build reusable engines once and configure modules — not rebuild screens manually for each module.
8.3
The Five Core Reusable Engines
Smriti OS is built on five universal engines that form the foundation of the entire system. Every module is a configuration of these engines, not a fresh build.
Engine
Component
Configuration Points
Benefit

&lt;AppGrid /&gt;
columns, permissions, APIs, filters, sort
One component used across all list screens
Universal Form Engine
&lt;AppForm /&gt; + JSON schema
fields array with name, type, validation
Auto-renders inputs, labels, errors from schema
Universal CRUD Engine
FastAPI Router Pattern
GET, POST, PUT, DELETE, LIST, EXPORT
Zero repeat code across all masters and transactions
Universal RBAC Engine
Permission Middleware
view, create, edit, delete, export per role
Same permission check everywhere — no rewrites
Universal Report Engine

filters, export, print, date range, pagination
Uniform report experience across all modules
8.4
Old Way vs Smart Way
Approach
Process
Result
Old Way — Manual Per Module
Design UI → Create Forms → Create APIs → Write Validations → Repeat for every module
Inconsistent UI, bugs per module, slow delivery, maintenance nightmare

8.5
What Cannot Be Reused — Business Logic Layer
Engine reuse applies to the UI and API scaffolding layer only. The real complexity of Smriti OS — what makes it genuinely valuable — lives in the business logic layer. These components are fully custom-coded and cannot be abstracted.
Complex Domain
Why Custom Code Is Required
POS / Billing Engine
Discounts, multi-tax, rounding rules, returns, holds — all domain-specific
Inventory Movement Engine
Stock in/out, inter-store transfers, batch tracking, FIFO/LIFO valuation
GST Rules Engine
CGST/SGST/IGST splits, HSN codes, e-invoicing, reverse charge — India-specific
Sync Engine
Delta sync, conflict resolution, offline queue management — highly stateful
Promotions &amp; Pricing Engine
Buy-X-get-Y, tier pricing, loyalty redemption — dynamic rule evaluation
8.6
Build 10 Engines — Not 100 Modules
The largest ERP systems in the world — SAP, Odoo, ERPNext, Zoho, Salesforce — all depend heavily on metadata, schema configurations, and reusable engines. They do not build hardcoded screens for every feature. Smriti OS follows this same proven strategy.
Build First — Core Engines
Then — Modules Become Configuration
Form Engine
Item Master = JSON schema config
Grid Engine
Customer Master = JSON schema config
Report Engine
Supplier Master = JSON schema config
RBAC Engine
All other masters = JSON schema config
Theme Engine
Transaction screens = business logic + engine config
Print Engine
Reports = report engine config + data query
CRUD Engine
New module = schema definition, not a new screen build
Core principle: You are not building 100 ERP modules. You are building 10 powerful reusable engines — and modules are their configurations.
8.7
Danger of Over-Generic Architecture
Reusable engines are powerful — but over-engineering them creates a different class of problems. Many ERP systems fail not because they lacked features, but because they became too abstract to maintain, debug, or extend.
Risk
Description
Excessive abstraction
Every workflow cannot be metadata-driven — some require explicit code
Reduced maintainability
Highly generic systems are harder to debug and understand
Performance overhead
Universal components can carry unnecessary overhead for simple cases
Edge case accumulation
Real deployments generate custom requirements that break generic assumptions
Reusable — Engines Handle This
Custom — Business Logic Requires This
Forms, Grids, Filters, Reports
POS Engine, GST Engine, Sync Engine
Permissions, Themes, Printing
Inventory Movement, Promotions, Pricing Rules
Rule: Reusable where possible. Custom where necessary. Never abstract purely for abstraction's sake.
8.8
Engine Override Strategy
Every real-world deployment eventually needs something slightly different. A rigid engine with no override path forces developers to modify core code, breaking it for everyone. Smriti OS uses a Layered Override Strategy.
Layer
Description
Who Owns It
Core Engine
Universal reusable engine — never modified directly
Architecture team

Schema and metadata defining fields, validations, standard behavior
Module developers
Module Overrides
Hooks that extend or replace specific engine behavior without touching core
Module developers
Client Extensions
Isolated per-client additions — live outside the core
Implementation partners
8.9
AI-Augmented Development Workflow
The reusable engine architecture of Smriti OS is designed to be AI-development-friendly. Because modules are defined as metadata and schema configurations rather than hand-coded screens, AI tools can generate repetitive implementation at high speed. This gives Smriti OS development velocity that would otherwise require a much larger team.
Who Owns It
Responsibility
Examples
Humans
Architecture and business logic decisions
Engine design, GST rules, sync conflict logic, pricing engine
Schemas
Define structure — fields, types, validations, relationships
Item Master schema, Customer schema, Report configuration
AI Tools
Generate repetitive implementation from schemas at high speed
Form generation, CRUD scaffolding, validation rules, API scaffolding, test generation
AI accelerates implementation speed. Humans control architecture correctness and business logic. Neither replaces the other.
8.10
Architectural Stance — Modular Monolith First
Smriti OS follows a Modular Monolith First strategy, evolving into service separation only where operational scale specifically requires it. This avoids premature microservices complexity, distributed system overhead, and unnecessary infrastructure cost — a mature architectural decision that keeps the system fast to build and easy to operate at early and mid-stage scale.
