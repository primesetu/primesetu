1.  Migration Philosophy
1.1  Inspired Schema Approach
SMRITI OS does not copy Shoper9's schema. It uses an Inspired Schema approach: field names and business logic are studied and preserved where practical, but the schema is redesigned from the ground up for PostgreSQL, multi-store cloud architecture, and offline sync. This eliminates legal risk and removes decades of technical debt from the legacy SQL Server single-store design.

2.2  Structural Audit Queries (Q1–Q8)
Q1 — All Tables with Row Counts
SELECT t.NAME AS TableName, p.rows AS RowCount
FROM sys.tables t
INNER JOIN sys.partitions p ON t.object_id = p.OBJECT_ID
WHERE p.index_id IN (0,1)
ORDER BY RowCount DESC;
-- Save: schema/row_counts.csv
Q2 — All Columns with Data Types
SELECT TABLE_NAME, COLUMN_NAME, DATA_TYPE, CHARACTER_MAXIMUM_LENGTH, IS_NULLABLE, COLUMN_DEFAULT
FROM INFORMATION_SCHEMA.COLUMNS
ORDER BY TABLE_NAME, ORDINAL_POSITION;
-- Save: schema/columns.csv
Q3 — Primary Keys
SELECT KU.TABLE_NAME, KU.COLUMN_NAME
FROM INFORMATION_SCHEMA.TABLE_CONSTRAINTS TC
INNER JOIN INFORMATION_SCHEMA.KEY_COLUMN_USAGE KU ON TC.CONSTRAINT_NAME = KU.CONSTRAINT_NAME
WHERE TC.CONSTRAINT_TYPE = 'PRIMARY KEY'
ORDER BY KU.TABLE_NAME;
-- Save: schema/primary_keys.csv
Q4 — Foreign Key Relationships
SELECT fk.name AS FK_Name, tp.name AS ParentTable, cp.name AS ParentColumn,
       tr.name AS ReferencedTable, cr.name AS ReferencedColumn
FROM sys.foreign_keys fk
INNER JOIN sys.foreign_key_columns fkc ON fk.object_id = fkc.constraint_object_id
INNER JOIN sys.tables tp ON fkc.parent_object_id = tp.object_id
INNER JOIN sys.columns cp ON fkc.parent_object_id = cp.object_id AND fkc.parent_column_id = cp.column_id
INNER JOIN sys.tables tr ON fkc.referenced_object_id = tr.object_id
INNER JOIN sys.columns cr ON fkc.referenced_object_id = cr.object_id AND fkc.referenced_column_id = cr.column_id
ORDER BY tp.name;
-- Save: schema/foreign_keys.csv
Q5 — Stored Procedures, Triggers, Functions, Views
SELECT name, type_desc FROM sys.objects WHERE type IN ('P','TR','FN','V')
ORDER BY type_desc, name;
-- Save: schema/procedures_triggers.csv
Q6 — Identity / Auto-Increment Columns
SELECT t.name AS TableName, c.name AS ColumnName
FROM sys.identity_columns c
INNER JOIN sys.tables t ON c.object_id = t.object_id
ORDER BY t.name;
-- Save: schema/identity_columns.csv
Q7 — Computed Columns
SELECT OBJECT_NAME(object_id) AS TableName, name AS ColumnName, definition
FROM sys.computed_columns ORDER BY TableName;
-- Save: schema/computed_columns.csv
Q8 — Non-Clustered Indexes
SELECT t.name AS TableName, i.name AS IndexName, i.type_desc, c.name AS ColumnName
FROM sys.indexes i
JOIN sys.index_columns ic ON i.object_id = ic.object_id AND i.index_id = ic.index_id
JOIN sys.columns c ON ic.object_id = c.object_id AND ic.column_id = c.column_id
JOIN sys.tables t ON i.object_id = t.object_id
WHERE i.is_primary_key = 0
ORDER BY t.name, i.name;
-- Save: schema/indexes.csv

4.  Type Mapping Decisions
4.1  UUID v4 vs UUID v7 — Documented Decision (FIXED in v3.0)
This was explicitly incorrect in v2.0. The decision between UUIDv4 and UUIDv7 has performance consequences at scale and must be a conscious, documented choice.
UUID Version
Characteristics
When to Use in SMRITI OS
UUID v4 (random)
Fully random — no time ordering. INSERT causes random B-tree page splits at scale. 16 bytes. Universally supported.
Master tables with low insert frequency: SmritiItem, CustomerMaster, SupplierMaster, CategoryMaster, TaxMaster, UserMaster, WarehouseMaster
UUID v7 (time-ordered)
Time-prefix + random suffix — monotonically increasing. Sequential inserts go to the same B-tree page. Dramatically better index locality on high-volume tables. Requires PostgreSQL 17+ native or uuid-ossp extension.
High-volume insert tables: SmritiSaleHdr, SmritiSaleDtl, SmritiPurchaseHdr, SmritiPurchaseDtl, stock_ledger, sync_events, audit_log, loyalty_points_ledger
Implementation Note — UUID v7
PostgreSQL 17 supports gen_random_uuid() for v4 only. For v7 use the 'pg_uuidv7' extension (github.com/fboulnois/pg_uuidv7) or generate in Python using the 'uuid7' package.
Python: from uuid7 import uuid7 — generates time-ordered UUID v7.
PostgreSQL extension: SELECT uuid_generate_v7(); — after installing pg_uuidv7.

6.3  Delete Policy — Masters vs Financial Transactions (FIXED in v3.0)
The v2.0 review correctly identified that applying is_deleted = TRUE to financial transaction records is a compliance risk. A deleted invoice must be traceable. The policy below defines which tables use soft delete and which use reversal entries.
Core Principle
Financial transaction records are NEVER soft-deleted. They are reversed by creating offsetting entries.
A cancelled invoice generates a credit note — not a delete flag.
This is both a GST compliance requirement and an audit integrity requirement.
Table Category
Tables
Delete Method
Reason
Master / Config tables
SmritiItem, CustomerMaster, SupplierMaster, CategoryMaster, BrandMaster, TaxMaster, UnitMaster, WarehouseMaster, UserMaster
is_deleted = TRUE (soft delete)
Masters can be deactivated — no financial impact. HO-pushed items: is_deleted only by HO_ADMIN. Store-created items: is_deleted by STORE_MANAGER.
Financial transaction headers
SmritiSaleHdr, SmritiPurchaseHdr, SmritiGRNHdr, StockTransferHdr
Reversal entry only — no delete
GST compliance — every invoice must remain traceable. Cancellation = new credit note record with negative amounts.
Financial transaction details
SmritiSaleDtl, SmritiPurchaseDtl, SmritiGRNDtl, StockTransferDtl
Follows header — never deleted independently
Detail rows are immutable once the header is finalized
Stock ledger
stock_ledger
Reversal entry only — no delete
Every stock movement is permanent. Errors corrected by opposing journal entry.
Payment records
payment_ledger, till_entries
Reversal entry only
Cash movements are immutable audit records
Loyalty points ledger
loyalty_points_ledger
Reversal entry only
Points earned and redeemed are permanent entries — reversals for returns
Sync events
sync_events, audit_log
No delete ever — append only
Compliance requirement — sync and audit history is immutable
Scheme / Config records
SmritiScheme, MenuConfig, ThemeConfig
is_deleted = TRUE + deactivate flag
Config can be deactivated but record retained for history
Store Manager Delete Permissions — AppForm Clarification
Store managers can soft-delete items they created locally (company_id matches, hq_pushed = FALSE).
Store managers CANNOT delete items pushed from Head Office (hq_pushed = TRUE — HO_ADMIN only).
This distinction must be enforced in both the RBAC engine and the FastAPI delete endpoint.
6.4  Index Strategy (NEW in v3.0)
Without a documented index strategy, every developer creates indexes based on instinct. The result is duplicate indexes, missing composite indexes, and degraded report performance at scale. The following indexes are required and must be created in the initial Alembic migration.
Index Rule
Never index a column that is a UUID primary key — PostgreSQL already indexes PKs.
Never create a single-column index on company_id alone — always composite with a business key.

6.5  Offline Conflict Resolution Model (NEW in v3.0)
Sync columns were present in v2.0 but there was no documented policy for what happens when two stores edit the same record offline simultaneously. This section defines the complete conflict resolution model.
The Problem
Store A and Store B both go offline at 10am. Store A edits Item X price to Rs.500. Store B edits Item X price to Rs.450. Both reconnect at 11am. Which value wins?

Transactions (bills, GRN, transfers)

Config (schemes, pricing, menus)

Same record edited at two stores simultaneously

Conflict Resolution Flow in integration.py
def resolve_conflict(local_record, incoming_record):
    # Same record edited at two locations simultaneously
    if incoming_record.sync_version > local_record.sync_version:
        # Incoming is newer — apply it
        apply_update(incoming_record)
        log_conflict(local_record, incoming_record, 'INCOMING_WINS')
    elif incoming_record.sync_version == local_record.sync_version:
        # Same version — use timestamp as tiebreaker
        if incoming_record.updated_at > local_record.updated_at:
            apply_update(incoming_record)
            log_conflict(local_record, incoming_record, 'TIMESTAMP_TIEBREAK')
        else:
            log_conflict(local_record, incoming_record, 'LOCAL_WINS')
    else:
        # Local is newer — keep local, discard incoming
        log_conflict(local_record, incoming_record, 'LOCAL_WINS')
def apply_update(record):
    record.sync_version += 1
    record.sync_status = 'synced'
    record.synced_at = datetime.utcnow()
    db.session.commit()
Eventual Consistency Boundary
Master records (items, pricing) are eventually consistent — stores may see different prices for up to 30 seconds after an HO push.
Transaction records (bills, GRN) are immediately consistent at their source store — they never change after finalization.
This boundary must be communicated to store managers during training.

1.  Migration Philosophy
1.1  Inspired Schema Approach
SMRITI OS does not copy Shoper9's schema. It uses an Inspired Schema approach: field names and business logic are studied and preserved where practical, but the schema is redesigned from the ground up for PostgreSQL, multi-store cloud architecture, and offline sync. This eliminates legal risk and removes decades of technical debt from the legacy SQL Server single-store design.

1.4  The Sovereign Database Layer
SMRITI OS uses sovereign.py as an intelligent proxy layer. SELECT KU.TABLE_NAME, KU.COLUMN_NAME

SELECT fk.name AS FK_Name,
       tp.name AS ParentTable, cp.name AS ParentColumn,
       tr.name AS ReferencedTable, cr.name AS ReferencedColumn
FROM sys.foreign_keys fk
INNER JOIN sys.foreign_key_columns fkc ON fk.object_id = fkc.constraint_object_id
INNER JOIN sys.tables tp ON fkc.parent_object_id = tp.object_id
INNER JOIN sys.columns cp ON fkc.parent_object_id = cp.object_id
   AND fkc.parent_column_id = cp.column_id
INNER JOIN sys.tables tr ON fkc.referenced_object_id = tr.object_id
INNER JOIN sys.columns cr ON fkc.referenced_object_id = cr.object_id
   AND fkc.referenced_column_id = cr.column_id
ORDER BY tp.name;

7.2  Migration Script Pattern — With Idempotency Guard (v2.0)
Every migration script must include an idempotency check. If the script crashes mid-wave and is re-run, it must not create duplicate records.
# Wave 2 example: ItemMaster → SmritiItem
# Idempotency guard: safe to re-run after crash
source_rows = mssql_session.query(Shoper9ItemMaster).all()
for row in source_rows:
    # IDEMPOTENCY CHECK — skip if already migrated
    existing = pg_session.query(SmritiItem).filter_by(
        legacy_id=row.ItemCode,
        company_id=COMPANY_UUID
    ).first()
    if existing:
        continue  # Already migrated — skip on re-run
    # Barcode placeholder if null (DQ1 resolution)
    barcode = row.ItemCode2 if row.ItemCode2 else f'GEN-{row.ItemCode}'
    target = SmritiItem(
        id             = uuid4(),
        legacy_id      = row.ItemCode,
        item_name      = row.ItemName,
        barcode        = barcode,
        mrp            = row.MRP,
        sale_rate      = row.SaleRate,
        hsn_code       = row.HSNCode,
        tax_master_id  = tax_uuid_map[row.TaxCode],
        category_id    = cat_uuid_map[row.CategoryCode],
        company_id     = COMPANY_UUID,
        store_id       = STORE_UUID,
        sync_status    = 'synced',
        created_by     = MIGRATION_USER_UUID,
        idempotency_key = uuid4(),
    )
    pg_session.add(target)
pg_session.commit()
print(f'Wave 2 complete — {len(source_rows)} rows processed')
